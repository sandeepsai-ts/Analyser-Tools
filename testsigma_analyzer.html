🚀Quick Setup
Three things to have ready before you start:
AI API Key Powers optimization suggestions. Get a free Gemini key at aistudio.google.com → Get API Key. Paste into the top-right AI API key field.
Testsigma API Key Generated from inside Testsigma — see steps below. Used as the Bearer token in all API calls. Paste into the Testsigma Bearer token field top-right.
Run ID Numeric ID of the test run. Found in the Testsigma URL when viewing a run.
Case Result ID Numeric ID of the specific test case result within that run.
How to generate your Testsigma API Key
Admin or Super Admin role required. Only users with Admin or Super Admin roles can generate API keys in Testsigma. If you don't have this role, ask your Testsigma admin to generate one for you.
1
Go to Settings → API Keys
In Testsigma, navigate to Settings in the left-side navbar → click API Keys.
2
Click "Generate new API Key"
Top-right corner of the API Keys page. A Generate API Key pop-up overlay will appear.
3
Name it and configure parallel settings
Enter a name (e.g. analyzer-tool). Optionally set the number of parallel test executions allowed for this key. Click Generate Key.
4
Copy the API Key
The key is displayed on screen. Click Copy API Key. This is the value you paste into the Testsigma Bearer token field in this tool — it is used as Authorization: Bearer <API_Key> in every API request.
The API key does not expire with your session — unlike a session token it persists until you delete it. Store it securely; treat it like a password.
Where to find Run ID & Case Result ID
1
Open a Test Run in Testsigma
Go to Test Runs → click any completed run.
2
Check the URL for the Run ID
The URL looks like:
app.testsigma.com/ui/runs/28760/details
That number is your Run ID.
3
Click into a specific test case result
The URL updates to include the Case Result ID:
…/test_case_results/287130/…
That number is your Case Result ID.
Both IDs change with every new run. Always use IDs from the specific run you want to analyse. The API Key however is reusable across runs.
Running the Analyser
1
Switch to the Analyser tab
Click ⚡ Analyse in the left rail.
2
Choose input method
Fetch Live — enter Run ID + Case Result ID, click Fetch All Pages. Auto-paginates all pages.
File / Paste — upload a JSON exported from Postman, or paste raw JSON.

3
Click Generate Insights

Read the matrix
Sort by any run column to find slowest steps. Red cells = >2s. PASS/FAIL badges come directly from Testsigma result data. Click Export Report for Excel.
⚙How the Analyser Works
The tool fetches Testsigma execution telemetry, normalises it, and sends it to an AI model in batches to generate per-step optimization recommendations.
Data Pipeline
Fetch Step 1
API pages are fetched from app.testsigma.com/api/v2/…/step_results with auto-pagination. Each page is 100 steps. A 210-step run = 3 pages. Pages are merged into one content array.
Normalise Step 2
Raw step objects are normalised into a flat schema. Loop iterations and step groups share the same stepId — these get composite keys like 93355:1, 93355:2 so no rows are lost.
Slim Payload Step 3
Only the fields the AI needs are sent: stepId, action, stepType, run*_sec, result. This reduces batch size by ~7x vs sending raw objects.
AI Batches Step 4
Steps are sent to the AI in batches of 25. Each batch gets a structured prompt asking for exactly 25 JSON objects back with suggestion and expected fields. Batches run sequentially with a 4s throttle to respect API rate limits.
Render Step 5
AI responses are merged back with Testsigma metadata (result, step type). The Performance Matrix table renders with sortable duration columns, PASS/FAIL badges, step type indicators, and slow-cell highlights.
Step Types
The AI is aware of step types and gives type-specific suggestions:
NLP_TEXT Standard automation steps. Suggestions focus on locator strategy and wait optimisation.
WHILE_LOOP Loop iterations. Each iteration gets its own row. Suggestions target loop exit conditions and polling intervals.
STEP_GROUP Reusable step groups. Suggestions target slow sub-step extraction and result caching.
API_STEP Direct API calls. Suggestions target timeout tuning and direct DB substitution.
Rate Limits & Fallback
If the AI API returns a 429 (rate limit), the tool reads the exact retry delay from the error message and waits before retrying — up to 2 retries per batch. If all retries fail, the step gets a fallback row (highlighted amber) with a generic suggestion. No steps are ever dropped.
Gemini free tier: 20 requests/min. A 210-step run = 9 batches. Expect ~2 minutes total with the 4s inter-batch throttle.
Multi-Run Comparison
Up to 3 runs can be loaded simultaneously. Each additional run adds a duration column. Percentage delta vs Run 1 is shown inline. The AI receives all run durations together and can identify variance-based issues across executions.
🔌Proxy Setup
Corporate networks block public CORS proxies. The fix is deploying your own in 3 minutes using a Cloudflare Worker — free, permanent, and only forwards to Testsigma.
Cloudflare Workers free tier: 100,000 requests/day, no credit card required. More than sufficient for a QA team.
Step 1 — Create a Cloudflare account
1
Go to cloudflare.com
Click Sign Up. Enter your email and a password. No credit card needed for the free plan.
2
Verify your email
Cloudflare sends a verification link. Click it to activate the account.
Step 2 — Create the Worker
1
Go to Workers & Pages
In the Cloudflare dashboard left sidebar, click Workers & Pages → Create.
2
Click "Create Worker"
Give it a name, e.g. ts-proxy. Click Deploy (deploys the default worker first — you'll replace the code next).
3
Click "Edit Code"
Select all the default code in the editor and replace it with the code below. Then click Deploy.
Worker Code — copy and paste this exactly
export default { async fetch(request) { // Only allow requests from your GitHub Pages domain const ALLOWED_ORIGIN = 'https://sandeepsai-ts.github.io'; const origin = request.headers.get('Origin') || ''; if (request.method === 'OPTIONS') { return new Response(null, { headers: corsHeaders(origin) }); } const url = new URL(request.url); const target = url.searchParams.get('url'); if (!target || !target.startsWith('https://app.testsigma.com/')) { return new Response('Only Testsigma URLs allowed', { status: 403 }); } const authHeader = request.headers.get('Authorization') || ''; const upstream = await fetch(target, { method: request.method, headers: { 'Authorization': authHeader, 'Content-Type': 'application/json' } }); const body = await upstream.text(); return new Response(body, { status: upstream.status, headers: { ...corsHeaders(origin), 'Content-Type': 'application/json' } }); } }; function corsHeaders(origin) { return { 'Access-Control-Allow-Origin': '*', 'Access-Control-Allow-Methods': 'GET, OPTIONS', 'Access-Control-Allow-Headers': 'Authorization, Content-Type', }; }
Update ALLOWED_ORIGIN to match your GitHub Pages URL if it differs from the one above. The worker only forwards requests to app.testsigma.com — all other URLs return 403.
Step 3 — Get your Worker URL and paste it in the tool
1
Copy the Worker URL
After deploying, Cloudflare shows your Worker URL:
https://ts-proxy.YOUR_NAME.workers.dev
2
Paste into the Proxy URL field
In the tool topbar, paste the URL into the Proxy URL (optional) field. The tool will use your proxy first, bypassing all public proxies.
3
Share with your team
Tell teammates to paste the same Worker URL into their Proxy URL field. It persists across sessions in the browser.
The Proxy URL is saved in sessionStorage automatically — you only need to paste it once per browser session.
